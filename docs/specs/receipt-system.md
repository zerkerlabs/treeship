# Receipt system, three tiers, one data model, agent first

**Status:** draft, not implemented
**Pairs with:** action/v2 mandate + effect (`packages/core/src/statements/action_v2.rs`, shipped v0.21), the paper document-of-record re-skin (PR #236, `design/receipt-and-token-spine`), session package manifest signing (`codex/receipt-report-redesign`, queued 0.21.1), the design token spine (`design/tokens.json`)
**Last updated:** 2026-07-21

## The shift

The receipt is the product. Everything else in Treeship exists to produce one and to let a third party check it. So the receipt is not one artifact rendered one way. It is a single signed data model rendered for three different audiences, in three different mediums, without ever letting the rendering claim more than the data supports.

The design principle is **agent first**: the receipt *is* the structured, machine-parseable record, and every human-facing rendering is a view over it. An AI agent, a CI job, a counterparty's verifier, or the same agent one week later must be able to parse the whole thing and reason about it, including what is proven, what is merely asserted, and what could not be checked. The honesty gradient (proven-here vs asserted vs not-verified, structural-pass vs pass) is a first-class, machine-readable property, not prose in a footer.

## The three tiers

| Tier | Name | Medium | Audience | Look |
|------|------|--------|----------|------|
| 1 | **Machine receipt** | terminal output + raw JSON, and a matching HTML view | developers, CI, other agents | dark, dense, monospace |
| 2 | **Human receipt** (document of record) | HTML generated by `treeship session report` | auditors, humans, print and share | light, engraved, serif |
| 3 | **Proofmark** (internal codename: glass) | web and social card | the public | light, glass |

The tiers map cleanly to medium and audience. Nothing is called "receipt" on the public surface: tier 3 is a **Proofmark**, the product; "glass" stays an internal design name only. The underlying data is a session receipt; the Proofmark is the surface that displays it.

The tiers are not competing designs. They are the same receipt for different readers. The token spine (below) makes them one visual family even across dark terminal, light paper, and web glass.

### Tier 1, the machine receipt (agent first, both renderings)

Two renderings of the same data:

- **Terminal.** What `treeship verify` and `treeship session report` print. Dark and dense is native to the terminal, not a style choice. This is the developer and CI surface, and it is what another agent scrapes.
- **HTML.** A dark, dense browser view of the same fields, for developers who want it in a page. Driven by the same token spine as tier 2 so the two are obviously one system.

The machine receipt is the upgrade of the existing `preview_template.html` and CLI output, extended to carry every primitive Treeship now produces (see the field inventory). It is the source of truth; tiers 2 and 3 are curated views of it.

### Tier 2, the human receipt (document of record)

The paper document-of-record re-skin (PR #236): light, engraved, Fraunces serif, the formal artifact `treeship session report` hands a human. It is the same sections as the machine receipt, presented for reading, printing, and sharing rather than parsing. When landing #236, **restore the Approval gates section** (human approval checkpoints) that the re-skin dropped; every other section survived the re-skin.

### Tier 3, Proofmark (the glass web card)

The live web and social surface (`treeship.dev/agents`, `/receipt/[id]`), already shipping. Rename it off "receipt" in the UI copy: it is a Proofmark. This is a labeling change on the existing surfaces, not a redesign.

## Agent-first field inventory

The machine receipt must surface every primitive, each with an explicit honesty label. Existing sections carry over; the new ones come from v0.21 and the memory-provenance spine.

**Carried over (already in the receipt today):** summary stats, agents, timeline (append-only event log), commands run, files changed, tool authorization (declared vs actual), financial-action evidence, security findings, **approval gates**, trust chain, verification checks, raw receipt, how-it-works.

**New, agent first (surface these as first-class sections):**

1. **Authority (mandate).** The grant each action ran under: grantor, scope, audience, expiry, delegation depth, revocation status. Answers "who authorized this."
2. **Effect and status ladder.** What each action touched (input, output, readback hashes, bytes, cost, side effects) and the reconciled confidence (verified, partial, ambiguous, unknown, not-verified) with the downgrade notes. The verdict separates operational confidence from cryptographic validity.
3. **Runtime identity.** Provider, model, tool-schema hash, system-prompt hash. Answers "what executed it."
4. **Witness map.** Independent corroborating observers per effect, and which ones a trusted authority vouched for.
5. **Blocked and refusal evidence** (`blocked.v1`). The negative space: what the agent was stopped from doing, and why.
6. **Quarantine gate.** Whether approvals cleared the memory-provenance check, and memory chain roots.
7. **Epoch identity** (pending, spec §2.8). A model:policy:runtime stamp per artifact.

Each field carries its proof status inline so a consuming agent knows exactly what it can rely on: a signed value, an asserted value, or a value that could not be verified from this surface.

## The token spine

`design/tokens.json` is the single source of truth for palette, type scale, and spacing, generated into `tokens.css`, `tokens.rs`, and `tokens.ts`. All three tiers read from it, so a color or scale change propagates to the terminal (via `tokens.rs`), the HTML receipts (via `tokens.css`), and the web Proofmark (via `tokens.ts`) at once. Dark terminal, light paper, and web glass are themes over one spine, not three unrelated palettes.
