# Capability Provenance: captured, not declared

**Status:** draft, not implemented
**Pairs with:** [agent capability cards](./agent-capability-cards.md), [agent resolver](./agent-resolver.md), discovery (`packages/cli/src/commands/discovery.rs`)
**Last updated:** 2026-06-24

## The gap this closes

A capability card today declares its tools by hand: `treeship attest card --tools file.*,db.query`. That string is *asserted*, an operator typed it, or worse, an agent emitted it. The card is signed and key-bound, so we know *who* made the claim, but the claim itself, "this agent can do X", is unverified. The entire registry rests on it.

This is the "the model told me" failure mode, and it is the difference between a registry of **claims** and a registry of **captured truth**. Capability provenance closes it: every declared capability carries the *source* of the claim, and verification surfaces it. Treeship stops asking you to trust the tool list and starts telling you where it came from.

## The principle

Identity and behavior are already captured deterministically: the keypair is generated by the machine, action receipts are hook-captured from real tool calls, never narrated. Capabilities are the one place a claim still sneaks in by hand. So apply the same rule: **a capability is only as trustworthy as its provenance, and the provenance must be reported, never assumed.**

## The provenance ladder

Each declared capability gets graded against two deterministic sources Treeship already has:

| Grade | Means | Source (already on disk / in the store) |
|---|---|---|
| **captured** | The tool is actually wired into the agent's harness. | discovery reads the agent's MCP / tool config (`mcp.json`, harness settings) |
| **exercised** | The capability is backed by real captured action receipts. | the `verify-capability` cross-check over signed action receipts |
| **declared** | Asserted only, neither wired nor ever exercised. | the operator's `--tools` string and nothing else |

A capability can be more than one (a wired tool that has also been used is `captured` + `exercised`). The strongest is `captured` (the machine sees it in the config); `exercised` is strong evidence of real use; `declared` is the honest label for everything else.

## What an honest card looks like

```jsonc
{
  "schema": "agent_card.v1",
  "agent": "agent://deployer",
  "capabilities": { "tools": ["file.*", "db.query", "deploy.prod"] },
  "capability_provenance": {
    "file.*":      { "grade": "captured",            "source": "harness:mcp.json#filesystem" },
    "db.query":    { "grade": "exercised",            "evidence": { "receipts": 14 } },
    "deploy.prod": { "grade": "declared" }            // wired nowhere, never used -> flagged
  }
}
```

`deploy.prod` is the interesting one: declared, but not in any config and never exercised. Today that capability is indistinguishable from the real ones. With provenance it is visibly weaker, and a reviewer (or the resolver) can treat it accordingly.

## Honest limits

- **`captured` proves the tool is *wired*, not that the agent is *confined* to it.** A wired tool the agent never calls is still a capability it *could* exercise; a tool absent from config the agent somehow calls would show up as `exercised` without `captured`, which is itself a useful signal. Confinement is enforcement, a runtime job, not something provenance claims.
- **`exercised` proves *use*, not the capability ceiling.** An agent that has only ever called 2 of its 10 declared tools is not proven incapable of the other 8; they are simply unbacked by evidence and labeled so.
- Provenance never *blocks* a card. It grades it. The honest-contract rule from the cards spec holds: report the strength, never overclaim.

## Schema

Add an optional `capability_provenance` object to `agent_card.v1`: a map from capability (the same string as in `capabilities.tools`) to `{ grade, source?, evidence? }`. Optional and additive, a card without it is treated as fully `declared` (today's behavior), so nothing breaks.

## Commands

- **`treeship attest card --from-harness`** builds the tool list from the agent's discovered MCP / harness config instead of `--tools`, and stamps each entry `captured` with its `source`. The deterministic-capture path: the capability set itself comes from observed wiring, not a flag.
- **`treeship verify-capability`** computes `exercised` from the action cross-check it already runs, merges it with any `captured` provenance on the card, and reports the breakdown: `capabilities: 1 captured, 1 exercised, 1 declared-only`.
- **`treeship resolve`** and the resolver bundle surface the same breakdown, so a resolved agent's capability grades travel over the network with it.

## Slices

1. **Provenance schema + `exercised` grading.** Add `capability_provenance` to `agent_card.v1` (additive). Have `verify-capability` and `resolve` compute and report the `exercised` grade from captured receipts (no new capture needed, it is already counting in-scope actions per tool). This alone moves every existing card from "all declared" to "declared vs evidence-backed."
2. **`--from-harness` capture.** Build the tool list from discovered MCP / harness config and stamp `captured` with `source`. This is the deterministic-capture path for *new* cards.
3. **Surface the full ladder** in `verify-capability`, `resolve`, and the resolver bundle, and (optionally) in the browser viewer via the existing WASM path.

## Open questions

1. **Capability granularity.** Harness config exposes MCP *servers* and tool *names*; cards declare exact tools or `family.*` globs. The mapping from "server `filesystem` is wired" to "the `file.*` family is captured" needs a defined rule (server-to-family map, or capture at the tool-name level where the config is that specific).
2. **Staleness.** Harness config is read at mint time; an agent reconfigured later has a stale `captured` grade. The `evidence_anchor` pattern applies, the card commits to the config it saw, and a re-mint refreshes it. Resolution should note the card's age.
3. **Cross-machine capture.** `captured` is read on the machine that mints the card. A card minted on a laptop carries that laptop's config provenance; this is consistent with the existing per-machine trust boundary, but the resolver should not imply the wiring still holds elsewhere.

## First slice to build

Slice 1: the `capability_provenance` schema field plus `exercised` grading in `verify-capability` and `resolve`. It is additive, needs no new capture machinery (the action cross-check already exists), and on day one it turns every card's capabilities from a flat list of claims into "evidence-backed vs declared-only", the first real step from a registry of claims to a registry of captured truth.
