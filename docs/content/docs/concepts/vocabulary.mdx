<Callout type="info">
Treeship uses nautical vocabulary deliberately. Every term maps to a concrete technical concept -- nothing is decorative.
</Callout>

| Term | Definition | Technical equivalent |
|------|-----------|---------------------|
| **Treeship** | A sovereign trust domain | Ed25519 keypair + config + local artifact store |
| **Captain** | Root human identity, warrant authority | The human who holds the ship key |
| **Crew** | Humans and agents aboard a Treeship | Actor URI registry in config.json |
| **Warrant** | Cryptographic authorization for one agent action | ApprovalStatement + nonce binding |
| **Passage** | Signed transfer of work between crew or Treeships | HandoffStatement |
| **Endorsement** | Second party verifies another artifact | EndorsementStatement |
| **Log entry** | A single signed record of something that happened | ActionStatement |
| **Ship's log** | Portable sealed record of a mission | .treeship bundle file |
| **Mission** | Named unit of work with start, checkpoints, sealed log | Bundle session + TREESHIP_MISSION env var |
| **Harbor** | Hosted infrastructure where Treeships dock and share | treeship.dev Hub + trust federation |
| **Fleet** | Collection of Treeships under one organization | Multi-Treeship context on one machine |
| **Berth** | Authenticated connection to the Harbor | dock_id + DPoP keypair |

## Backward compatibility

Old commands still work as aliases:

| Old | New | Both work |
|-----|-----|-----------|
| `treeship attest action` | `treeship attest log` | Yes |
| `treeship attest approval` | `treeship attest warrant` | Yes |
| `treeship dock login` | `treeship berth login` | Yes |
| `treeship dock push` | `treeship berth push` | Yes |
