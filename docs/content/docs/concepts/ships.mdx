A ship is the core trust domain in Treeship. It consists of a single Ed25519 keypair and a local artifact store. Every artifact signed within a ship shares the same root of trust.

## What a ship is

A ship is three things:

1. **A keypair.** One Ed25519 private key (encrypted at rest) and one public key. The private key signs every artifact. The public key is how verifiers confirm your signatures.
2. **An artifact store.** A local directory (`~/.treeship/`) containing all signed artifacts, the Merkle tree, and configuration.
3. **An identity boundary.** Everything signed by this ship shares the same trust root. If you trust the ship's public key, you trust every artifact it signed.

```bash
treeship init
```

```
Ship initialized
  name: my-workspace
  key:  key_9f8e7d6c (ed25519, encrypted at rest)
```

## One ship, many agents

A ship can have many agents working inside it. Each agent is identified by a URI (`human://alice`, `agent://deployer`), but agents do not hold their own keys. They all act under the ship's key.

```
Owner (human)
+-- Ship key (Ed25519)
    |-- agent://researcher  -- signs artifacts with ship key
    |-- agent://deployer    -- signs artifacts with ship key
    |-- human://alice       -- signs artifacts with ship key
    +-- human://bob         -- signs artifacts with ship key
```

This means every artifact from this ship can be verified with a single public key, regardless of which agent created it. The agent URI tells you who acted. The ship key tells you the artifact is authentic.

## When to create a new ship vs. add a hub connection

This is the key decision most users face:

**Add a hub connection** when you want the same identity to publish artifacts to a different Hub workspace. Your artifacts carry the same signature either way. Hub connections just control where artifacts appear.

Use cases for adding a hub connection:
- Publishing to both a personal workspace and a work workspace
- Sharing artifacts with a client through their workspace
- Maintaining separate public and internal workspaces

**Create a new ship** when you need a completely separate identity with its own keypair. The two ships share nothing -- different keys, different artifact stores, different trust roots.

Use cases for creating a new ship:
- Personal projects vs. work projects that should not share a trust root
- Separate organizations or companies
- Isolated environments where cross-contamination of trust is unacceptable

<Callout type="info">
A good rule of thumb: if you would use the same SSH key, use the same ship. If you would use different SSH keys, use different ships.
</Callout>

## Ship identity on Hub

When you attach to Hub, your ship's public key is registered. If the same public key appears in multiple hub connections, Hub knows they belong to the same ship. This is how Hub links workspaces to identities without requiring accounts or emails.

## Key management

The ship's private key is encrypted at rest using the system keychain (macOS Keychain, Linux Secret Service, or a passphrase fallback). The key never leaves your machine. Hub never receives any private key -- only the public key is transmitted during the device flow.

```bash
treeship keys list
```

```
KEY ID          TYPE      CREATED
key_9f8e7d6c   ed25519   2026-03-26T21:00:00Z (encrypted at rest)
```
