A Treeship is the core trust domain. It consists of a single Ed25519 keypair and a local artifact store. Every artifact signed within a Treeship shares the same root of trust.

## What a Treeship is

A Treeship is three things:

1. **A keypair.** One Ed25519 private key (encrypted at rest) and one public key. The private key signs every artifact. The public key is how verifiers confirm your signatures.
2. **An artifact store.** A local directory (`~/.treeship/`) containing all signed artifacts, the Merkle tree, and configuration.
3. **An identity boundary.** Everything signed by your Treeship shares the same trust root. If you trust the Treeship's public key, you trust every artifact it signed.

```bash
treeship init
```

```
Treeship initialized
  name: my-workspace
  key:  key_9f8e7d6c (ed25519, encrypted at rest)
```

## One Treeship, many agents

A Treeship can have many agents working inside it. Each agent is identified by a URI (`human://alice`, `agent://deployer`), but agents do not hold their own keys. They all act under the Treeship key.

```
Owner (human)
+-- Treeship key (Ed25519)
    |-- agent://researcher  -- signs artifacts with Treeship key
    |-- agent://deployer    -- signs artifacts with Treeship key
    |-- human://alice       -- signs artifacts with Treeship key
    +-- human://bob         -- signs artifacts with Treeship key
```

This means every artifact from your Treeship can be verified with a single public key, regardless of which agent created it. The agent URI tells you who acted. The Treeship key tells you the artifact is authentic.

## When to create a new Treeship vs. add a hub connection

This is the key decision most users face:

**Add a hub connection** when you want the same identity to publish artifacts to a different Hub workspace. Your artifacts carry the same signature either way. Hub connections just control where artifacts appear.

Use cases for adding a hub connection:
- Publishing to both a personal workspace and a work workspace
- Sharing artifacts with a client through their workspace
- Maintaining separate public and internal workspaces

**Create a new Treeship** when you need a completely separate identity with its own keypair. The two Treeships share nothing -- different keys, different artifact stores, different trust roots.

Use cases for creating a new Treeship:
- Personal projects vs. work projects that should not share a trust root
- Separate organizations or companies
- Isolated environments where cross-contamination of trust is unacceptable

<Callout type="info">
A good rule of thumb: if you would use the same SSH key, use the same Treeship. If you would use different SSH keys, use different Treeships.
</Callout>

## Treeship identity on Hub

When you attach to Hub, your Treeship's public key is registered. If the same public key appears in multiple hub connections, Hub knows they belong to the same Treeship. This is how Hub links workspaces to identities without requiring accounts or emails.

## Key management

The Treeship's private key is encrypted at rest using the system keychain (macOS Keychain, Linux Secret Service, or a passphrase fallback). The key never leaves your machine. Hub never receives any private key -- only the public key is transmitted during the device flow.

```bash
treeship keys list
```

```
KEY ID          TYPE      CREATED
key_9f8e7d6c   ed25519   2026-03-26T21:00:00Z (encrypted at rest)
```
