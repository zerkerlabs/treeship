Treeship's trust model has one axiom: the signatures are the trust.

## No trusted third party

A Treeship artifact is valid if its Ed25519 signature verifies against a known public key. Validity does not depend on:

- Treeship's servers being online
- The Hub being reachable
- Any certificate authority
- Any token or session

<Callout type="info">
The Hub is convenience infrastructure for sharing and discovery. It cannot forge a passing verification result.
</Callout>

## Ships, agents, and hub connections

Treeship's trust architecture has three layers:

### Layer 1: Agents

Agents are the actors that perform work inside a ship. Each agent is identified by a URI:

- `human://alice` -- a person
- `agent://coder` -- an AI agent
- `human://approver` -- someone who issues approvals

Agents do not hold their own keys. They act under the ship's key.

### Layer 2: Ships

A ship is the trust domain. One ship means one Ed25519 keypair and one local artifact store. Every artifact created within a ship is signed by the same key, regardless of which agent created it.

```
Owner (human)
└── Ship key (Ed25519)
    ├── Signs all artifacts
    ├── Signs approvals
    └── Signs handoffs
```

The owner holds the private key, encrypted at rest.

### Layer 3: Hub connections

A hub connection is a named link from your ship to a Hub workspace. Hub connections are how your trust domain connects to the outside world.

Think of hub connections like tmux sessions. Your local ship stays the same. Hub connections just control where you push and which workspace your artifacts appear in.

```
Ship (one keypair, one artifact store)
├── hub: default  -> hub.treeship.dev/alice/personal
├── hub: acme     -> hub.treeship.dev/acme-corp/projects
└── hub: client-x -> hub.treeship.dev/clientx/deliverables
```

Key properties of hub connections:

- Adding or removing a hub connection never changes your ship key
- Artifacts are identical regardless of which hub connection you push them through
- Each hub connection has its own DPoP keypair for authenticating with Hub
- Most developers only need one hub connection

### When to create a new ship vs. add a hub connection

**Add a hub connection** when you want the same identity to publish artifacts to a different Hub workspace. Your artifacts carry the same signature either way -- hub connections just control visibility.

**Create a new ship** when you need a separate identity with its own keypair. For example, a personal ship and a work ship, or separate ships for separate projects that should not share a trust root.

## Chain integrity

Every artifact contains a `parentId` field pointing to the previous artifact in the chain. Verification walks this chain recursively. A broken link (missing parent, tampered content) causes the entire chain to fail.

## Approval binding

Approvals use a nonce-binding model. The approver signs an approval containing a random nonce. The agent includes that nonce in its action. At verification time, the verifier checks:

```
action.approvalNonce == approval.nonce
```

This is enforced in the Rust core. There is no way to skip the check.

## Offline verification

```bash
treeship verify release.treeship
```

<Callout type="success">
This works on an airplane, in a secure facility, or on a machine that has never connected to the internet. The .treeship bundle contains everything needed: artifacts, signatures, and the public key.
</Callout>
